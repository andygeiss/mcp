# MCP Spec Conformance — Narrative Draft

This document is the Q2 draft of the conformance narrative for `github.com/andygeiss/mcp`. It wraps the mechanically-rendered audit fragment at [`docs/spec-coverage.txt`](../spec-coverage.txt) in procurement-readable framing. The Q4 publication will canonicalize what is drafted here.

If this narrative ever disagrees with the rendered fragment, the rendered fragment wins — it is generated by `make spec-coverage` from the live `protocol.Clause` registry and is the source of truth for clause IDs, normative levels, sections, and backing test names.

## Project framing

`mcp` is a single-binary Go implementation of the Model Context Protocol (MCP), version `2025-11-25`. It communicates over stdin/stdout using newline-delimited JSON-RPC 2.0. There is no HTTP, no WebSocket, no parallel transport.

The implementation rests on three load-bearing constraints. Each is documented in [`docs/architecture.md`](../architecture.md#load-bearing-constraints) and surfaces directly in the conformance evidence:

- **Stdlib-only.** No external `go.mod` dependencies beyond the standard library. CI tooling (golangci-lint, GitHub Actions, Codecov) is exempt infrastructure, not project dependencies.
- **Sequential dispatch.** The server advertises `experimental.concurrency.maxInFlight: 1` to signal sequential request handling to clients. This invariant is reflected in every handler-side test.
- **Wire discipline.** Stdout carries protocol bytes only. Stderr is structured `slog.JSONHandler` exclusively. Tests inject `io.Reader`/`io.Writer` so transport behavior is observable byte-for-byte.

These constraints are not preferences — they are the foundation that the audit moat extends. See the [Architecture overview](../architecture.md) for the full rationale, including the bidi reader-split design [ADR-003](../adr/ADR-003-bidi-reader-split.md) that pins the four invariants (AI7 cancel chain, AI8 typed errors, AI9 capability gate, AI10 no-progress-during-outbound).

## Evidence model

Conformance claims in this project are not free-text. They are **clauses** — typed records registered in code, adjacent to the tests that back them.

The mechanism is small enough to summarize completely:

1. The `protocol.Clause` type lives in [`internal/protocol/spec_clauses.go`](../../internal/protocol/spec_clauses.go). It records a clause `ID`, normative `Level` (`MUST` / `SHOULD` / `MAY`), `Section` reference, and the function values of the backing tests.
2. Tests register clauses from `_test.go` `init()` blocks adjacent to the assertions they make. Registration captures the test function pointer; the renderer resolves the function name via `runtime.FuncForPC` so the rendered fragment is byte-traceable to the running test.
3. `make spec-coverage` aggregates per-package fragments ([`docs/spec-coverage.protocol.txt`](../spec-coverage.protocol.txt) and [`docs/spec-coverage.server.txt`](../spec-coverage.server.txt)) into the canonical [`docs/spec-coverage.txt`](../spec-coverage.txt) via the aggregator at [`cmd/spec-coverage/`](../../cmd/spec-coverage). The cross-package aggregation pattern is documented in [`docs/development-guide.md`](../development-guide.md).

The narrative below cites [`docs/spec-coverage.txt`](../spec-coverage.txt) as the audit evidence for every clause it names. If `make spec-coverage` is re-run after new clauses land, this narrative is regenerated by hand against the new snapshot. The renderer is canonical; this narrative is its prose wrapper.

As of 2026-05-03, the registry contains 11 unique clause IDs across 6 spec areas, 13 backing test references, all at normative level `MUST`. The `SHOULD` and `MAY` levels are reserved — the registry will broaden as later stories add weaker-than-MUST conformance evidence.

## Per-area clause inventory

The clauses below are grouped by spec area. Within each area, every clause names its ID, normative level, section reference, and backing test function(s) — pulled verbatim from [`docs/spec-coverage.txt`](../spec-coverage.txt). Test names retain their full Go import path so they can be located in the source tree without grep gymnastics.

### `decode` — input safety at the wire

The decode area covers structural limits enforced at JSON unmarshal time. The single clause here pins the project's defense against attacker-supplied deeply nested JSON: the decoder rejects nesting beyond a configured cap before any handler dispatch, returning a parse error rather than risking a stack-overflow panic.

### `elicitation` — outbound bidi capability gating

The elicitation area covers `elicitation/create` requests sent from a tool handler back to the client. Both clauses encode the capability-honesty rule (Invariant AI9, [ADR-003](../adr/ADR-003-bidi-reader-split.md)): a server must not issue an outbound elicitation request unless the client advertised the corresponding capability during `initialize`. The first clause pins the failure mode (typed `CapabilityNotAdvertisedError` with zero side effects); the second pins the round-trip when the gate passes.

### `jsonrpc` — id preservation

The JSON-RPC area covers the §5 Response object rule that response `id` must be echoed back exactly as sent — including type. A request id sent as a JSON string must come back as a string; sent as a number must come back as a number. The single clause here is backed by two tests pinning each id type independently.

### `progress` — token discipline and bidi non-interference

The progress area is the most populous, covering four MUST rules around `notifications/progress` from tool handlers. Token plumbing is observable: a request that supplies a `_meta.progressToken` MUST reach the handler context (extract clause); a handler that calls `Progress.Report` without a token MUST NOT emit (no-token clause); a token MUST round-trip preserving its JSON type, string or number (preserve-type clause). Invariant AI10 forbids progress notifications during an active outbound bidi request to prevent interleaving with client-bound traffic on the shared writer.

| `MCP-2025-11-25/progress/MUST-not-emit-during-outbound` | MUST | AI10 — no progress notifications during outbound bidi requests | `github.com/andygeiss/mcp/internal/server.Test_Progress_Report_With_OutboundActive_Should_Drop` |

| `MCP-2025-11-25/progress/MUST-preserve-token-type` | MUST | Q6 progress-token passthrough discipline | `github.com/andygeiss/mcp/internal/server_test.Test_ProgressReport_With_StringToken_Should_PreserveType`, `github.com/andygeiss/mcp/internal/server_test.Test_ProgressReport_With_NumberToken_Should_PreserveType` |

## Refresh discipline

This narrative is regenerated by hand after each `make spec-coverage` run that adds, removes, or renames clauses. The publication target is Q4 2026; until then the file carries `status: 'draft — Q4 publication target'` in its YAML front matter.

The mechanical fragment at [`docs/spec-coverage.txt`](../spec-coverage.txt) is the canonical evidence. This narrative wraps it; it does not duplicate the registry's contents in a third location. If the two ever drift, the rendered file wins — fix the narrative, never the renderer's output by hand.

When new clauses land:

1. The PR introducing the clause registers it via a `protocol.Clause` entry in an `init()` block adjacent to the backing test (NFR4 in the Q2 epic plan).
2. `make spec-coverage` regenerates the canonical fragment.
3. This narrative is updated in a follow-up commit (or the same PR, if scope permits) so the prose tracks the evidence.

Adding `SHOULD` or `MAY` clauses does not require a structural change to the narrative — extend the relevant per-area table and note the level in the inline prose where it carries weight.
