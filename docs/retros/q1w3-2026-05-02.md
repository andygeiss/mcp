## What Didn't Go Well / Friction Points

### 1. Build-tag complexity on the server fragment test

Amelia (Developer): Story 3.1 hit a subtle build-tag wrinkle. Story 2.2's Q5 clauses register from `internal/server/integration_test.go` (tagged `//go:build integration`). For the server fragment to capture those clauses, `internal/server/spec_coverage_test.go` ALSO had to be tagged integration. Net effect: drift detection on the server fragment moves out of `make check`'s default test path and into CI's separate integration job. Documented in completion notes, but it's a real ergonomics hit — local `go test ./...` doesn't catch server-fragment drift.

Mitigation: the new aggregator unit test `Test_run_With_CommittedFragments_Should_MatchCommittedAggregate` runs under default `go test ./...` and catches aggregate drift even when fragment drift would slip past locally. Partial protection.

### 2. AC5 PARTIAL: shared `initRequest` constant ignored

Mary (Business Analyst): Story 3.2 AC5 said "every bidi-test `initialize` request payload" — strictly, that includes the synctest AI9 negative-path tests, which use the shared `handshake()` → `initRequest` constant. Adding `protocolVersion` there would touch dozens of non-bidi tests. Scope-discipline call: leave it for a separate sweep. Documented as deferred. The PARTIAL is the auditor's accurate reading of the literal AC, not a slip.

**Lesson:** for next wave's stories, pre-flight the AC text against shared constants/helpers. "Every X" language should explicitly call out shared-resource exclusions if they exist.

### 3. The "test owns its drainage inline" exception in progress_test.go

Amelia (Developer): The AI10 progress test couldn't use the new `drainBidi` helper because `drainBidi` calls `assertCleanStderr` after drain — which conflicts with the AI10 test's expected single `progress_dropped_during_outbound` warn record on stderr. The test owns its drainage inline (`stdinW.Close + stdoutR.Close + <-done`) and skips the clean-stderr check. This is a deliberate divergence from the helper pattern, documented in a comment, but it's a sign the helper's contract is too rigid (drainage and stderr-checking shouldn't be entangled).

**Future cleanup candidate:** split `drainBidi` into `drainBidi(...) error` (returns the run error, no stderr check) and a separate `assertCleanStderr` call. The current test files already use both that way; the helper just bundles them for convenience.

### 4. Aggregate drift would not have been caught by CI without the new aggregator unit test

Edge Case Hunter (review finding): Story 3.1's Makefile target drift-detects the aggregate locally, but neither `make check` (which runs `make spec-coverage`? — no, it doesn't) nor CI's integration job invokes the aggregator step. Without the new `Test_run_With_CommittedFragments_Should_MatchCommittedAggregate` unit test (added during Story 3.1's review patches), aggregate drift would only be caught when a developer manually ran `make spec-coverage`. The unit test closes the gate.

**Lesson:** any drift-detection mechanism needs a CI gate, not just a developer convenience target. Apply this to future audit-extension work.

## Key Insights

### A. The triad's value is finding latent runtime bugs, not surface-level style

Story 3.2's catches were textbook: cleanup ordering deadlocks, panic-leak goroutines, swallowed errors, buffer races. None of these would have triggered locally during the implementer's testing. The triad's independence (three reviewers with three different methodologies) is what surfaces them. **Do not skip the auditor pass on test-hygiene work** — the conventional wisdom that "test code is less critical" is wrong; test code that hides assertions is worse than no test at all.

### B. Discipline > Features (W1 insight, reaffirmed for the third wave)

W1 said the moat is the audit, not the features. W2 used the audit infrastructure. W3 finished building it AND swept the test-hygiene debt that accumulated under W2's velocity. Three waves in, the strategic frame is intact: every TS/Python MCP server can ship features faster, but none of them have these three things together (W1 audit moat + W2 protocol-surface adds + W3 test-discipline hardening).

### C. AC text is contract; pre-flight it against shared resources

Story 3.2's AC5 PARTIAL is the canonical example of an AC that read accurately at draft time and missed a subtle scope question (shared `initRequest`). Future stories that touch "every X" should explicitly enumerate the X's or call out shared-resource exclusions in the AC itself.

### D. Repurpose-don't-replace held under execution

`cmd/spec-coverage/main.go` was repurposed from a compile-checked stub to a real aggregator. The original commit-checked goal of "binary stays compile-clean" is preserved as a side effect (it's still a `main` package that builds under `go build ./...`). The resistance to "delete and rewrite" paid off — the binary's history-of-existence is intact, and the file already had the right import path/package shape.
