## What Went Well

### 1. The plan absorbed scope discoveries without breaking

Mary (Business Analyst): The locked plan held its order through execution. Items shipped in roughly the planned sequence (1, 2, 3, 4, 5, 6-Q60, 7) with one principled deviation: S3 was carved off mid-flight when implementation cost (`Tool.InputSchema` restructure across ~30 test sites) outweighed value (allocs win on a non-hot path). That carve-off was *captured*, not silently dropped — `deferred-work.md:39-47` documents the decision and the revisit triggers.

Winston (Architect): The plan's "Don't do as a discrete PR" section (M4 state-machine table, A1 middleware chain) protected the team from a class of refactor-first mistakes. Three states didn't justify a table; we kept the dispatch flat as planned.

### 2. Spec carve-off discipline kept PR scope tight

Amelia (Developer): M1 split into M1a (safety floor) + M1b (value tuning + ADR-004) when the spec doc exceeded the 1600-token ceiling. The result: M1a was a focused review (typed error + non-fatal branch + panic guards + RED tests) and M1b was a second focused review (value tuning + envelope raise + ADR-004 + per-tool v2 plan). Reviewers got two small clean PRs instead of one sprawling one. `deferred-work.md` made the split visible and revisit-able.

### 3. Externalized correctness pressure paid off immediately

Amelia (Developer): Fuzz_Server_E2E's first run surfaced `id: true` (and other invalid id types) being echoed back in error responses — a JSON-RPC 2.0 §5 violation. Found and fixed in the same PR (`0f0a331`). This is the highest possible ROI fuzz outcome: write the harness, get a real bug for free, ship the fix bundled. The plan called this "every later PR runs against it" — that promise is now load-bearing.

### 4. Capability honesty (R2) compounded with M2+M3

John (PM): R2 (capability honesty) is small but lands in the right place after M2 (`error.data`) and Q41 (stderr startup banner). An operator running the server now gets: a banner with pid + go_version + protocol version, errors with structured data, request_id auto-injected in logs, AND honest capability advertisement. The 3am debugging story changed from "misery" to "tedious" — exactly what M2+M3+Q41 were named for.

Sally (UX): This bundle is the user-facing win of Wave 1. Most of the other items are invisible to consumers — security floors, fuzz harnesses, schema discipline. Operator dignity is the one Wave 1 deliverable an integrator *feels* on day one. Lead the next changelog / release notes with it, not with M1 or Fuzz_E2E. "Errors now carry structured `data`, requests now carry an auto-injected `request_id`, startup writes a banner with pid/go_version/protocol — your 3am self will thank past you" is a better opening line than "decode-time structural limits added."

### 5. Stdlib-only constraint never wobbled

Winston (Architect): Seven PRs touching protocol, codec, dispatch, schema, capabilities, version negotiation, and golden tests. Zero new `go.mod` entries. The constraint that looked binding looked load-bearing in retrospect — every PR forced first-principles solutions (counting reader instead of `io.LimitReader`, `slices.Contains` instead of pulling a set library, `sync.Once` patterns instead of dependency-injection frameworks).

**Reframe (party-mode amendment):** stdlib-only is *a* moat, not *the* moat. The load-bearing constraint is **sequential dispatch** (`maxInFlight: 1`) — that's what lets the codec, the state machine, the bidi peer, and the goldens stay simple enough to audit. Stdlib-only is the visible discipline; sequential dispatch is the architectural decision that makes the visible discipline cheap. Worth naming explicitly before Wave 2 considers any concurrency surface.

### 6. Goldens replaced JSON Schema review with JSON Schema diff review

Paige (Tech Writer): Q60 is small in code and large in discipline. Every future PR that mutates a tool input struct now surfaces the wire-shape change in a reviewable diff. The schema-evolution conversation moved from "does this look right?" to "do we want this change?" That's a category improvement.

## What Didn't Go Well / Friction Points

### 1. No formal sprint-status.yaml — BMM tooling can't read state

Amelia (Developer): The plan doc (`q1-plan-2026-04-29.md`) is doing the job of an epic-stories file plus a sprint-status file. That worked fine for a 2-day, 7-PR push by a single developer + agent. It will not work for Wave 2 if the dependency graph gets denser or if `bmad-create-story` / `bmad-sprint-status` need to drive selection.

**Evidence:** During this very retrospective preparation, I (Amelia, in the form of the bmad-help / sprint-status / create-story sequence) gave Andy a wrong answer twice — first in sprint-status, where I told him R5 was still "remaining" when it had already shipped on `32a4d15`, then again in create-story when I almost wrote a phantom story for shipped work. The tooling that should have prevented that mistake isn't wired up.

**Root cause:** No `sprint-status.yaml`. The skills key off it. They have nothing to read.

### 2. The "S3+Q60" plan-list bundle was misleading

Mary (Business Analyst): Item #6 in the locked plan was "S3 + Q60". In execution, only Q60 shipped. The deferred-work entry captures *why* S3 was carved (net cost > value), but the strategic frame in the plan still says: "Q1 Wave 1 (M1+Fuzz+R5+**S3**+Q60) all landed AND ≥1 enterprise procurement question lands in issues" as the A2 dossier revisit trigger. With S3 deferred, that trigger condition is technically unmet.

**Implication:** Either the trigger needs to be redefined to count Q60 as the wave-1 deliverable for item #6 (likely, since the plan's own deferred-work entry says Q60 delivers the "conscious-evolution discipline" that mattered), or A2 dossier publication waits longer than planned.

### 3. M1a coverage gaps surfaced post-merge

Amelia (Developer): The edge-case-hunter review of M1a found 4 coverage gaps (in `deferred-work.md:26-35`). None are bugs — production behavior is correct in each case — but they are *invariant-hardening* tests that should land before any decoder-adjacent work in Wave 2.

**Specifically deferred:**
1. In-flight × structural-limit interaction test
2. Mid-flight handler-abandonment timer test (`synctest`)
3. Alternating object/array nesting test
4. Tool-timeout vs structural-limit distinguishability test

**Implication:** Bundle into a small follow-up PR before Wave 2 touches the decoder or dispatch paths.

### 4. The plan called for a "CI matrix `for v := range supported`" — the matrix is degenerate

Winston (Architect): R5 (`32a4d15`) shipped the negotiation infrastructure, and the unit tests use `for v := range protocol.SupportedVersions`. But `SupportedVersions` has exactly one entry today, so the matrix is a list of one. The strategic value (graceful upgrade across multiple revisions) only kicks in when MCP publishes the next revision and a second entry gets prepended.

**Not a defect** — the plan explicitly framed R5 as "insurance against the next MCP spec rev" — but it's worth noting that the matrix's *visible* exercise is pending an external trigger.

## Key Insights

### A. Discipline > Features (the moat point made tangible)

The brainstorming source's strategic frame — "the moat is the audit, not the features" — survived contact with execution. Wave 1 didn't ship a single new MCP method or surface area. What it shipped was: a security floor (M1), continuous adversarial coverage (Fuzz_E2E), surface reduction (E-sweep), operator dignity (M2+M3+Q41), upgrade insurance (R5), schema diff discipline (Q60), and protocol honesty (R2). Every TS/Python MCP server can ship features faster. None of them have these seven things together.

### B. Carve-offs are first-class artifacts

`deferred-work.md` ended Wave 1 with three substantive entries (M1b shipped, M1a coverage gaps, S3 design rationale). That doc is now a load-bearing piece of the planning system — it's where decisions go to *survive* rather than disappear. Worth treating as a permanent artifact, not session ephemera.

### C. The 1600-token spec ceiling is a real constraint

Hit twice in Wave 1: once forcing M1 → M1a + M1b, once implicit in why S3 didn't get its own spec doc (it would have needed one to capture the ~30-test-site refactor scope). The ceiling is doing useful work: it forces decisions to be either "small enough to ship" or "split into smaller pieces." Worth keeping.

### D. The party-mode review caught real misjudgments

The plan doc says: "Three follow-up rounds resolved the contested calls." The party-mode hits that mattered:
- **A2** rescoped from "Top 1% engineering pick" to "discipline now, publication deferred to 2026 Q4"
- **M4** rescoped from "highest-confidence theme #1" to "opportunistic refactor, not discrete PR"
- **Format for spec registry** changed from "tagged comments + grep CI" to "typed Go registry with function-pointer Tests"

All three would have been wrong calls without the multi-agent stress-test. Worth running again before Wave 2 commits.
