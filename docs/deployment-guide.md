# Deployment Guide

## Overview

The MCP server is a single static binary with no runtime dependencies. It communicates over stdin/stdout and requires no network configuration, database, or service discovery. The binary is designed to be invoked by MCP clients (e.g., Claude Code) as a subprocess.

## Building

### Local Build

```bash
# Build with version tag
go build -ldflags "-X main.version=$(git describe --tags --always --dirty)" ./cmd/mcp/

# Build without version (defaults to "dev")
go build ./cmd/mcp/

# Verify
./mcp --version
```

### Build Flags

| Flag | Description |
|---|---|
| `-X main.version=<version>` | Embed version string (printed by `--version`) |

The version defaults to `"dev"` when not set via ldflags.

## Release Process

Releases are automated via [GoReleaser](https://goreleaser.com/) triggered by pushing a semantic version tag:

```bash
git tag v1.0.0
git push origin v1.0.0
```

### Release Pipeline (`release.yml`)

The release workflow performs:

1. **Checkout** with full history (`fetch-depth: 0`) for proper versioning
2. **Setup Go** from `go.mod` version
3. **Install cosign** (Sigstore) for keyless signing
4. **Install syft** (Anchore) for SBOM generation
5. **Run GoReleaser** with `--clean` flag

### Release Artifacts

| Artifact | Description |
|---|---|
| `mcp_<version>_darwin_amd64.tar.gz` | macOS Intel binary |
| `mcp_<version>_darwin_arm64.tar.gz` | macOS Apple Silicon binary |
| `mcp_<version>_linux_amd64.tar.gz` | Linux x86_64 binary |
| `mcp_<version>_linux_arm64.tar.gz` | Linux ARM64 binary |
| `checksums.txt` | SHA-256 checksums for all archives |
| SBOM | Software Bill of Materials (via Syft) |
| Cosign signature | Keyless signing of checksums via Sigstore |

### Supply Chain Security

- **Cosign keyless signing** — checksums signed via Sigstore without managing private keys
- **SBOM generation** — Software Bill of Materials generated by Syft for every release
- **All GitHub Actions pinned to full commit SHAs** — prevents tag-based supply chain attacks (e.g., `actions/checkout@de0fac2e...` instead of `@v4`)
- **Zero external Go dependencies** — eliminates Go module supply chain risk entirely

## CI/CD Pipelines

### CI Pipeline (`ci.yml`)

**Triggered on:** push to `main`, pull requests.
**Concurrency:** Cancels in-progress runs for the same branch (avoids wasted compute).

| Job | Timeout | Description | Dependencies |
|---|---|---|---|
| `build` | 5 min | Verifies `go.mod` is tidy, runs `make build` | none |
| `test` | 10 min | Tests with race detector on 3 OS matrix (macOS, Ubuntu, Windows); coverage uploaded to Codecov (except Windows) | build |
| `lint` | 5 min | golangci-lint with 45+ linters, zero issues required | build |
| `vulncheck` | 5 min | govulncheck v1.1.4 for known vulnerabilities | build |
| `fuzz` | 10 min | Protocol decoder fuzzing for 2 minutes | build |
| `integration` | 10 min | Integration tests with race detector (`-race -tags=integration`) | build |
| `bench` | 10 min | Benchmarks with 20% regression threshold; compares against `testdata/benchmarks/baseline.txt` using `benchstat` | build |
| `ci-ok` | — | Final aggregation job — fails if any job failed | all |

**Test matrix:**
- macOS (latest)
- Ubuntu (latest) — with coverage upload
- Windows (latest)

### Security Pipelines

| Pipeline | File | Triggers | Description |
|---|---|---|---|
| **CodeQL** | `codeql.yml` | PR, push to main, weekly (Monday 06:00 UTC) | Static security analysis for Go using GitHub's CodeQL engine |
| **OpenSSF Scorecard** | `scorecard.yml` | Push to main, weekly (Monday 06:00 UTC), manual dispatch | Supply chain security assessment; uploads SARIF results |
| **Nightly Fuzz** | `fuzz.yml` | Daily 03:00 UTC, manual dispatch (configurable fuzztime) | Extended fuzzing of 4 targets |

### Nightly Fuzz Pipeline (`fuzz.yml`)

Runs 4 fuzz targets sequentially with `set -euo pipefail`:

| Target | Package | Default Duration |
|---|---|---|
| `Fuzz_Decoder_With_ArbitraryInput` | `internal/protocol` | 5 min |
| `Fuzz_Server_Pipeline` | `internal/server` | 5 min |
| `Fuzz_ValidateInput_With_ArbitraryInput` | `internal/tools` | 5 min |
| `Fuzz_ValidatePath_With_ArbitraryInput` | `internal/tools` | 5 min |

Manual dispatch allows overriding `fuzztime` (default: `5m`).

### OSS-Fuzz Integration

The project is integrated with [Google OSS-Fuzz](https://github.com/google/oss-fuzz) for continuous fuzzing. The `oss-fuzz/` directory contains:

| File | Purpose |
|---|---|
| `Dockerfile` | Build container based on `base-builder-go` |
| `build.sh` | Compiles the `Fuzz_Decoder_With_ArbitraryInput` target |
| `project.yaml` | Project metadata: libFuzzer engine, address sanitizer |

**Local testing:**
```bash
docker build -f oss-fuzz/Dockerfile -t mcp-fuzz-test .
```

### Benchmark Regression Detection

CI compares benchmark results against `testdata/benchmarks/baseline.txt`:

```bash
# Run benchmarks (6 iterations for statistical significance)
go test -bench=. -count=6 -benchmem ./internal/... > current.txt

# Compare against baseline
benchstat testdata/benchmarks/baseline.txt current.txt
```

Regressions exceeding 20% cause CI failure.

**Updating the baseline** after confirmed improvements:
```bash
go test -bench=. -count=6 -benchmem ./internal/... > testdata/benchmarks/baseline.txt
```

## MCP Client Configuration

### Claude Code

The `.mcp.json` file configures Claude Code to use the server:

```json
{
  "mcpServers": {
    "mcp": {
      "command": "go",
      "args": ["run", "./cmd/mcp/"]
    }
  }
}
```

### Generic MCP Client

Any MCP client can invoke the server by spawning the binary as a subprocess:

```bash
# The server reads JSON-RPC 2.0 from stdin and writes responses to stdout
./mcp
```

**Protocol handshake:**
1. Client sends `initialize` request with capabilities
2. Server responds with server info, protocol version, and capabilities
3. Client sends `notifications/initialized` notification
4. Server transitions to ready state — all methods now available

**Example session:**
```jsonl
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"capabilities":{}}}
{"jsonrpc":"2.0","method":"notifications/initialized"}
{"jsonrpc":"2.0","id":2,"method":"tools/list","params":{}}
{"jsonrpc":"2.0","id":3,"method":"tools/call","params":{"name":"echo","arguments":{"message":"hello"}}}
```

## Dependency Management

- **Runtime dependencies**: None (stdlib only) — verified by `claudemd_test.go`
- **CI toolchain** (not project dependencies): golangci-lint, govulncheck, benchstat, cosign, syft, goreleaser
- **Dependabot**: Configured in `.github/dependabot.yml` for two ecosystems:
  - **GitHub Actions**: Weekly schedule, max 5 open PRs
  - **Go modules**: Weekly schedule, max 5 open PRs
- **Action pinning**: All GitHub Actions are pinned to full commit SHAs for supply chain security

## Security Policy

- Only the latest minor release receives security fixes
- Vulnerability reports via GitHub Security Advisories (private)
- Response SLA: acknowledgment within 72 hours, assessment within 7 days, fix within 30 days
- Scope: MCP server binary (`cmd/mcp`) only — `cmd/init` is not security-critical
- No formal bug bounty program
